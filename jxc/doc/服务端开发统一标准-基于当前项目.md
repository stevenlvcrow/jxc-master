# 服务端开发统一标准（基于当前项目）

## 1. 适用范围
- 适用于 `jxc/src/main/java/com/boboboom/jxc/**` 和 `jxc/src/test/java/com/boboboom/jxc/**`。
- 本文档只描述当前项目实际采用的服务端开发规范，不使用额外架构术语作为前提。

## 2. 基本原则
- 真实数据优先：正式接口必须以数据库真实数据为准，不允许用内存假数据冒充正式结果。
- 服务端闭环：权限、作用域、归属、可操作性都必须在服务端校验，不能依赖前端控制。
- 作用域显式：任何集团/门店/平台相关操作都必须显式带作用域，不允许默认回退到其他作用域。
- 结果可验证：每个新增功能都必须能通过测试或最小联调证明其真实生效。

## 3. 分层职责

### 3.1 `interfaces/rest`
- 只负责接收请求、做协议适配、返回响应。
- 可以做基础参数校验、字符串 trim、枚举入口值校验。
- 可以读取登录上下文，但不能直接写数据库。
- 禁止直接注入和调用 Mapper。
- 禁止在 Controller 内写复杂查询拼装、批量写入、权限判断、事务编排。

### 3.2 `application/service`
- 负责业务流程编排、事务边界、状态流转、权限判断、资源归属判断。
- 负责把请求对象转换成可持久化对象，或者组织多个仓储完成完整动作。
- 负责返回 controller 需要的结果对象。
- 允许依赖 repository，不允许把 SQL 逻辑散落到 controller。

### 3.3 `domain/repository`
- 只定义仓储能力，不放 Web 逻辑，不放 SQL 细节。
- 仓储接口只表达业务需要的数据读写能力。

### 3.4 `infrastructure/persistence`
- 负责 Mapper、Repository 实现、DO、SQL、查询对象。
- 复杂查询优先放在 Mapper XML 或 repository 实现里收口。
- 任何跨多个表的组合查询，都应当在这一层或 application/service 层收口，不要散在 controller。

## 4. Controller 标准
- 一个 controller 只服务一个清晰职责域。
- controller 方法只做四件事：
  - 接收参数
  - 基础校验
  - 调用 service
  - 返回响应
- controller 体积超过约 `600` 行时必须开始拆分。
- controller 超过约 `800` 行不得继续堆功能，必须拆分或下沉逻辑。
- controller 不得包含：
  - Mapper 调用
  - QueryWrapper 组装
  - 批量持久化
  - 作用域权限判断
  - 复杂状态流转

## 5. Service 标准
- service 负责完整业务动作，不负责 HTTP 协议。
- 写操作必须明确事务边界。
- service 内部可以调用多个 repository，但不要把 repository 当成简单 DAO 直接堆到 controller。
- 业务规则统一放 service，避免复制到多个 controller。
- `requireXxx`、`ensureXxx`、`normalizeXxx` 这类能力应当优先收敛到 service 或专门辅助类。

## 6. 数据访问标准
- 优先通过 repository 访问数据。
- controller 不允许直接依赖 mapper。
- service 尽量不要直接散落 mapper，公共查询和持久化规则应下沉到 repository。
- repository 只做数据访问，不判断业务权限。
- 查询结果必须按业务需要明确排序，不依赖数据库默认顺序。

## 7. 作用域与权限标准

### 7.1 作用域格式
- `orgId` 统一使用以下形式：
  - 空值：平台作用域
  - `group-{groupId}`
  - `store-{storeId}`
- 非法格式必须直接失败，不得静默降级。

### 7.2 权限判断
- 平台管理员、集团管理员、门店账号的权限判断必须以服务端关系数据为准。
- 禁止通过用户名、手机号、默认 ID 等硬编码规则特判管理员。
- 任何读写接口都必须校验：
  - 登录态
  - 作用域
  - 资源归属
  - 操作能力

### 7.3 读接口规则
- 作用域型读接口禁止“当前作用域无数据时回退平台数据”。
- 请求了哪个作用域，就只在该作用域内查。
- 如果该作用域没有数据，返回空集或明确业务错误。

### 7.4 批量接口规则
- 批量修改、批量删除、批量导出必须先按当前作用域筛选命中集。
- 命中数量和请求数量不一致时，必须整批失败。
- 禁止静默跳过越权 ID 后返回成功。

## 8. 请求、响应与异常
- 请求对象统一放在 `interfaces/rest/request`。
- 响应对象统一放在 `interfaces/rest/response` 或局部 `View` 类型。
- 成功响应统一使用项目现有 `CodeDataResponse`。
- 业务错误统一抛 `BusinessException`。
- 未登录或登录失效统一按项目现有认证异常处理。
- 面向前端的错误信息必须是可理解的中文业务语义，不能直接暴露 SQL、表名、堆栈。

## 9. 命名标准
- `Controller`：接口层。
- `Service`：业务流程层。
- `Repository`：数据访问封装。
- `Request`：入参对象。
- `Response` / `View`：出参对象。
- `DO`：数据库映射对象。
- `Snapshot`：业务查询结果对象。
- `Support`：纯辅助类，只能做上下文、格式化、纯工具逻辑，不能直接碰持久化。

## 10. 事务标准
- 写接口必须显式标注事务。
- 读接口默认不加事务，除非有明确一致性要求。
- 一个事务只覆盖一个完整业务动作，不跨越无关模块。
- 禁止“先删全量再按权限补写”这类高风险流程。

## 11. 日志与审计
- 敏感信息不得写入日志：
  - 密码
  - token
  - 完整证件信息
- 高风险操作必须记录关键审计字段：
  - 操作者
  - 目标 ID
  - 作用域
  - 操作结果

## 12. 测试标准
- 新增业务能力至少要有一种测试：
  - service 测试
  - repository 测试
  - controller 集成测试
- 新增或修改权限、作用域、菜单、角色相关功能时，必须覆盖：
  - 成功路径
  - 参数非法
  - 对象不存在
  - 权限不足
  - 越权访问
  - 非法 `orgId`
- 角色菜单、机构树、作用域查询等配置型功能，必须有测试证明真实生效。

## 13. 建议的重构顺序
1. 收 controller 里的 mapper 直连。
2. 收 controller 里的业务判断和查询拼装。
3. 把重复查询沉到 repository。
4. 给关键业务补测试。
5. 再处理更大的接口和更复杂的组合查询。

## 14. 当前项目的落地要求
- `interfaces/rest` 只保留接口适配，不保留持久化访问。
- `application/service` 负责业务流程，不保留重复的 controller 逻辑。
- `infrastructure/persistence` 负责数据访问实现和 SQL 收口。
- 新增功能默认按 `Controller -> Service -> Repository` 的方式实现。
- 存量代码如果暂时没法一次性改完，至少不能继续新增同类坏味道。
