# 服务端开发统一标准（基于当前项目）

## 1. 文档目标
- 本文档用于统一 `jxc` 服务端开发方式，重点解决当前项目走查中已经出现的实际问题：
  - 菜单权限“可配置但不生效”。
  - 主数据在租户作用域为空时错误回退到平台数据。
  - 控制器直接堆砌业务逻辑，类体积失控，权限边界难以复核。
  - 部分模块仍处于“内存假数据 + 局部落库”的过渡状态，却以正式接口形式暴露。
  - 测试能启动但不能证明关键业务链路正确。
- 本文档不再只强调分层名词，而是把“权限、作用域、真实数据、可测试性、可发布性”列为强制标准。
- 适用范围：`jxc/src/main/java/com/boboboom/jxc/**` 与 `jxc/src/test/java/com/boboboom/jxc/**`。

## 2. 技术与运行基线
- Java：`21`
- Spring Boot：`4.0.5`
- MyBatis-Plus：`3.5.16`
- PostgreSQL：项目当前运行库版本
- Redis：项目当前运行缓存
- 构建验证：`mvn test`

新增依赖、升级框架、修改运行方式时，必须在变更说明中同步写明兼容性影响、回滚方式和验证范围。

## 3. 总体设计原则

### 3.1 真实数据优先
- 正式暴露给前端的业务接口必须基于真实持久化数据。
- 禁止新增“进程内内存 Map/列表 + 局部数据库同步”的正式业务实现。
- 允许临时 stub 的场景仅限：
  - 明确标注为临时演示接口；
  - 不接入正式菜单和正式路由；
  - 有明确下线时间和替换负责人。

### 3.2 权限必须在服务端闭环
- 前端菜单、按钮显隐、路由守卫都不是安全边界。
- 任何读写接口都必须在服务端完成：
  - 身份确认
  - 作用域确认
  - 资源归属确认
  - 操作能力确认

### 3.3 作用域不能隐式降级
- 请求了某个集团/门店作用域，就只能在该作用域内查。
- 如果该作用域下没有数据，返回空集或明确错误。
- 禁止因为“该作用域没数据”而自动回退到平台数据。

### 3.4 配置必须真实生效
- 角色、菜单、机构、用户关系等配置类接口，保存结果必须影响实际运行行为。
- 禁止出现“写库成功，但运行时查询完全不使用这些配置”的伪闭环。

## 4. 分层架构标准

### 4.1 标准职责
- `interfaces`：接收请求、做协议适配、读取鉴权上下文、返回响应。
- `application`：流程编排、事务边界、权限校验编排、跨仓储协同。
- `domain`：核心业务规则、领域模型、不依赖 Web 和 SQL 细节。
- `infrastructure`：DO、Mapper、Repository、外部系统访问。
- `common/config`：全局异常、日志、拦截器、统一配置。

### 4.2 强制要求
- 新增业务功能默认走 `Controller -> ApplicationService -> Repository` 路径。
- 禁止继续新增“大型 Controller 直连多个 Mapper”的实现。
- 存量直连代码允许维护，但满足以下任一条件时必须拆到 `application`：
  - 超过 3 个 Mapper 协同
  - 包含权限/范围判断
  - 包含批量写入或删除
  - 包含角色、菜单、组织关系计算

### 4.3 控制器规模限制
- 单个 Controller 超过约 `600` 行时必须开始拆分。
- 超过约 `800` 行不得继续堆功能，必须拆控制器或下沉应用层。

## 5. 接口与返回体标准

### 5.1 路由规则
- 业务接口统一使用 `/api/<context>/...`
- 新功能禁止再新增 `/auth/...`、`/org/...` 等非统一前缀接口。
- 存量非统一前缀接口视为待迁移接口，不再作为规范基线。

### 5.2 返回体规则
- 新接口统一使用：
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```
- 成功码统一使用 `0`。
- 历史兼容接口可继续兼容 `200`，但新增接口禁止再引入新的成功码风格。
- 同一模块内部禁止同时新增两种成功返回结构。

### 5.3 错误信息规则
- 面向前端的 `message` 必须是可理解的中文业务语义。
- 禁止把 SQL、表名缺失、连接串、堆栈、运维指令直接返回给前端。
- 环境初始化问题应记录日志，对前端返回统一业务可理解提示。

## 6. 参数、校验与异常标准

### 6.1 入参定义
- 请求对象统一放在 `interfaces/rest/request` 或 `application/command`。
- 禁止在 Controller 方法参数上堆大量裸字段，复杂请求必须封装对象。

### 6.2 参数校验
- 必填：`@NotBlank/@NotNull`
- 长度：`@Size`
- 格式：`@Pattern`
- 枚举型字段优先做显式值校验，不允许散落魔法字符串。

### 6.3 异常分层
- 可预期业务错误：`BusinessException`
- 未登录/登录失效：`UnauthorizedException`
- 统一由 `GlobalExceptionHandler` 收口
- Controller 禁止自己 `try/catch` 后返回半成功结果

## 7. 认证与登录态标准

### 7.1 鉴权入口
- 受保护接口统一由 `AuthInterceptor` 进入。
- 新增业务接口时，必须同步确认拦截器已覆盖对应路径。

### 7.2 Token 规则
- Token 只允许从以下位置解析：
  - `Authorization: Bearer <token>`
  - `X-Auth-Token`
- AccessToken 与 RefreshToken 必须是两个明确语义的概念。
- 禁止继续使用“同一个 token 同时作为 accessToken 和 refreshToken”的实现。

### 7.3 登录态存储
- 登录态统一走 `TokenService + Redis`。
- 登录会话必须包含最小必要信息：
  - `userId`
  - 显示名
  - 登录时间
  - 如后续需要，可扩展能力摘要，但不得放入敏感信息。

## 8. 权限与作用域标准

### 8.1 平台管理员判定
- 平台管理员只能通过角色/关系表判定。
- 禁止通过用户名、账号字符串、手机号、默认 ID 等硬编码规则判定管理员身份。
- 禁止前后端各自维护一套“admin 特判”。

### 8.2 机构作用域规则
- 统一使用 `orgId` 表示请求作用域，仅允许三种形式：
  - 空值：平台
  - `group-{groupId}`
  - `store-{storeId}`
- 非法格式必须直接失败，不得降级为平台。

### 8.3 作用域访问规则
- 平台管理员：可访问任意作用域。
- 集团管理员：可访问本集团及下属门店。
- 门店管理员：仅可访问本门店。
- 非平台账号请求平台作用域时，必须直接拒绝。

### 8.4 资源归属校验规则
- 多租户资源的查询、更新、删除必须同时带 `scope_type + scope_id`。
- 禁止对多租户资源直接使用裸 `selectById/updateById/deleteById` 作为最终写路径。
- 正确顺序必须是：
  1. 解析请求作用域
  2. 校验操作者是否有该作用域权限
  3. 在该作用域内查询资源
  4. 对命中的资源进行写操作

### 8.5 读接口禁止回退
- 作用域型读接口禁止“当前作用域无数据时回退平台层”。
- 这条规则适用于：
  - 供应商
  - 供应商分类
  - 物品分类
  - 单位
  - 统计类型
  - 后续所有租户主数据

### 8.6 批量接口规则
- 批量删除、批量修改、批量导出必须先按当前作用域筛出命中集。
- 请求数量与命中数量不一致时必须整批失败。
- 禁止静默跳过越权 ID 后返回成功。

## 9. 菜单、角色与权限配置标准

### 9.1 菜单下发规则
- 当前用户菜单必须基于真实关系计算：
  - 用户角色关系
  - 角色菜单关系
  - 菜单状态
  - 请求作用域
- 禁止继续使用纯硬编码 SQL 分支直接拼菜单树，绕开 `role_menu_rel`。

### 9.2 配置类接口标准
- 角色菜单关系保存后，必须影响：
  - 当前菜单查询
  - 页面可达性
  - 相关操作能力
- 如果只是“保存成功但当前运行逻辑未消费”，该接口视为未完成。

### 9.3 菜单查询验收要求
- 菜单查询必须覆盖以下场景：
  - 平台管理员
  - 集团管理员
  - 门店账号
  - 无权限账号
  - 非法 `orgId`
- 必须证明菜单差异来自真实权限关系，而不是字符串用户名特判。

## 10. 数据建模与持久化标准

### 10.1 多租户主数据强制字段
- 需要按平台/集团/门店隔离的主数据表，必须具备：
  - `scope_type`
  - `scope_id`
  - 审计字段
  - 必要唯一约束
  - 必要索引

### 10.2 唯一约束规则
- 作用域内唯一的数据，唯一键必须带上作用域字段。
- 禁止使用全局唯一键导致跨租户互相冲突。

### 10.3 假数据禁令
- 正式业务 Controller 中禁止初始化大批量演示数据作为默认列表。
- 正式业务接口中禁止返回“伪造总数、伪造 ID、伪造时间”的演示记录。

### 10.4 建表与变更流程
- 任何新增业务在联调前，开发者必须先执行对应 SQL。
- 执行后必须完成最小验证：
  - 至少一次插入
  - 至少一次查询
  - 至少一次唯一约束或作用域校验验证
- 提测说明中必须包含：
  - 表变更清单
  - SQL 执行结果
  - 最小验证结论

## 11. Repository 与 SQL 标准

### 11.1 Repository 规则
- Repository 负责聚合型持久化与查询封装。
- Controller 不允许直接组合多个 Mapper 完成复杂业务。

### 11.2 SQL 规则
- 复杂查询写在 Mapper XML。
- 查询必须有明确排序。
- 分页 SQL 必须稳定，禁止依赖默认顺序。
- Mapper XML 中若涉及权限和作用域判断，必须能被单独测试验证。

### 11.3 状态与枚举
- 状态、角色编码、作用域类型等公共常量必须集中定义。
- 禁止在多个控制器和 SQL 中散落同义魔法值。

## 12. 事务标准
- 写接口必须显式 `@Transactional`
- 读接口如存在多步一致性读取，建议 `readOnly = true`
- 先鉴权、再校验、后写入
- 禁止“先删全量再按权限补写”的危险流程

## 13. 日志与审计标准
- 接口访问日志统一通过现有日志拦截能力收口。
- 敏感信息不得写入日志：
  - 密码
  - token
  - 完整身份证明信息
- 高风险操作必须记录关键审计字段：
  - operatorId
  - targetId
  - scopeType/scopeId
  - 操作结果

## 14. 测试标准

### 14.1 基本要求
- 所有新增业务能力至少包含一种测试：
  - 应用层测试
  - SQL/Mapper 测试
  - 关键控制器集成测试

### 14.2 强制覆盖场景
- 成功路径
- 参数非法
- 对象不存在
- 权限不足
- 越权访问
- 非法 `orgId`

### 14.3 配置类功能额外要求
- 角色菜单保存后，必须有测试证明菜单查询结果发生变化。

### 14.4 测试可用性要求
- `mvn test` 必须可作为有效信号。
- 如果当前测试依赖特定 schema、大小写表名或初始化脚本，必须把前置条件固化到测试环境，不允许靠人工脑补。

## 15. 提测与发布验收清单
1. 新增功能未引入新的正式假数据接口。
2. 无新增 Controller 直连多个 Mapper 的复杂实现。
3. 所有作用域型接口已做到“显式作用域、显式拒绝、无平台回退”。
4. 平台管理员判定不依赖用户名或账号字符串。
5. 菜单权限、角色菜单关系已形成真实闭环。
6. 批量接口已做命中全集校验，无部分成功越权风险。
7. 数据表与唯一约束已按作用域建模。
8. 至少执行一次 `mvn test`，且通过与本次改动相关的测试。
9. 至少验证一组跨集团或跨门店越权失败用例。
10. 提测说明中包含数据库变更和真实联调结果。

## 17. 存量整改计划

### 17.1 禁止继续复制的反模式
- 已发现存在问题的实现不得再作为新功能复制模板：
  - 用户名硬编码管理员判定
  - 菜单硬编码下发
  - 作用域为空时回退平台数据
  - 内存假数据混合正式持久化
  - 超大 Controller 持续堆叠

### 17.2 优先整改顺序
1. 菜单权限真实生效
2. 去除作用域回退
3. 收敛正式接口中的假数据实现
4. 拆分超大 Controller 并补测试

### 17.3 路由统一整改要求
- 后续接口整改统一收口到 `/api/<context>/...`
- 存量 `/auth/...`、`/org/...` 接口不得继续扩展新能力，只允许迁移、兼容和下线
- 前后端联调时优先改造调用方到统一前缀接口
