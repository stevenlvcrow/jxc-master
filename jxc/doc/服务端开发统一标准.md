# 服务端开发统一标准
# 适用于 jxc 项目所有后端开发，所有新功能必须严格遵守
# 代码复审（CR）以此文档为唯一依据，违规项必须修改

## 1. 适用范围
# 约束项目主源码与测试代码范围
- 适用于 `jxc/src/main/java/com/boboboom/jxc/**` 和 `jxc/src/test/java/com/boboboom/jxc/**`。
- 本文档只描述当前项目实际采用的服务端开发规范，不使用额外架构术语作为前提。

## 2. 基本原则
# 项目开发核心底线，所有逻辑必须遵循
- 真实数据优先：正式接口必须以数据库真实数据为准，不允许用内存假数据冒充正式结果。
- 服务端闭环：权限、作用域、归属、可操作性都必须在服务端校验，不能依赖前端控制。
- 作用域显式：任何集团/门店/平台相关操作都必须显式带作用域，不允许默认回退到其他作用域。

## 3. 分层职责
# 四层架构职责边界，禁止越层写逻辑

### 3.1 `interfaces/rest`
# 接口层：仅处理 HTTP 交互，不写业务/数据逻辑
- 只负责接收请求、做协议适配、返回响应。
- 可以做基础参数校验、字符串 trim、枚举入口值校验。
- 可以读取登录上下文，但不能直接写数据库。
- 禁止直接注入和调用 Mapper。
- 禁止在 Controller 内写复杂查询拼装、批量写入、权限判断、事务编排。

### 3.2 `application/service`
# 业务层：核心业务逻辑、事务、权限、流程编排
- 负责业务流程编排、事务边界、状态流转、权限判断、资源归属判断。
- 负责把请求对象转换成可持久化对象，或者组织多个仓储完成完整动作。
- 负责返回业务数据给 Controller，不处理 HTTP 协议。
- 允许依赖 repository，禁止直接使用 Mapper 或拼装查询条件。

### 3.3 `domain/repository`
# 仓储接口层：只定义数据读写能力，无具体实现
- 只定义仓储能力，不放 Web 逻辑，不放 SQL 细节。
- 仓储接口只表达业务需要的数据读写能力。

### 3.4 `infrastructure/persistence`
# 数据持久层：所有 SQL、查询、Mapper、数据实现收口
- 负责 Mapper、Repository 实现、DO、SQL、查询对象。
- 复杂查询优先放在 Mapper XML 或 repository 实现里收口。
- 任何跨多个表的组合查询，都应当在这一层收口。

## 4. Controller 标准
# 接口层写法规范：轻量化、无业务、无数据逻辑
- 一个 controller 只服务一个清晰职责域。
- controller 方法只做四件事：接收参数、基础校验、调用 service、返回响应。
- controller 体积超过约 600 行时必须开始拆分，超过 800 行不得继续堆功能。
- 禁止包含：Mapper 调用、QueryWrapper 组装、批量持久化、作用域权限判断、复杂状态流转。

## 5. Service 标准
# 业务层写法规范：纯业务、禁 SQL、禁 HTTP 协议
- service 负责完整业务动作，不负责 HTTP 协议。
- 写操作必须明确事务边界。
- 禁止直接注入或使用 Mapper。
- 禁止拼装 QueryWrapper、手动拼接 SQL。
- 业务规则统一收敛，禁止复制到多个地方。

## 6. 数据访问标准
# 数据操作规范：必须走仓储，禁止裸用 Mapper
- 优先通过 repository 访问数据。
- controller 不允许直接依赖 mapper。
- service 禁止直接使用 mapper，所有查询下沉到 repository。
- repository 只做数据访问，不判断业务权限。
- 查询结果必须明确排序，不依赖数据库默认顺序。

## 7. 作用域与权限标准
# 权限与数据隔离核心规范，防止越权/越域

### 7.1 作用域格式
# 固定作用域格式，非法格式直接拒绝
- `platform`：平台作用域
- `group-{groupId}`：集团作用域
- `store-{storeId}`：门店作用域
- 非法格式必须直接失败，不得静默降级。

### 7.2 权限判断
# 所有接口必须校验，禁止信任前端传参
- 权限判断必须以服务端关系数据为准，禁止硬编码特判管理员。
- 任何读写接口必须校验：登录态、作用域、资源归属、操作能力。
- 作用域必须从服务端安全上下文获取，禁止信任前端传入的 orgId。

### 7.3 读接口规则
# 数据隔离：不自动回退、不越域查询
- 请求哪个作用域，只在该作用域内查询，不自动回退平台数据。
- 无数据返回空集或明确业务错误。

### 7.4 批量接口规则
# 批量操作安全：越权即全失败，不静默跳过
- 批量操作必须先按当前作用域筛选，越权则整批失败。
- 禁止静默跳过越权数据后返回成功。

## 8. 请求、响应与异常
# 统一入参、出参、异常格式
- 请求对象统一放在 `interfaces/rest/request`。
- 响应对象统一放在 `interfaces/rest/response` 或 `View` 包。
- 成功响应统一使用 `CodeDataResponse`，业务错误统一抛 `BusinessException`。
- 错误信息为业务语义，不暴露 SQL、表名、堆栈。

## 9. 命名标准
# 统一命名规则，见名知意
- `Controller`：接口层
- `Service`：业务流程层
- `Repository`：数据访问封装
- `Request/Response/View`：入参/出参对象
- `DO`：数据库映射对象
- `Snapshot`：业务查询结果对象
- `Support`：纯辅助类，不直接操作持久化

## 10. 事务标准
# 事务使用安全规范
- 写操作必须显式标注事务。
- 读接口默认不加事务。
- 一个事务只覆盖一个完整业务动作，禁止长事务、跨模块事务。
- 禁止“先删全量再补写”等高风险流程。

## 11. 日志与审计
# 日志安全与关键操作留痕
- 禁止写入日志：密码、token、完整证件信息。
- 高风险/列表/分页/树形查询必须记录：操作者、作用域、关键字、数据量级。

## 12. 数据库脚本标准
# SQL 脚本规范，避免启动失败
- 只允许使用 Spring 可直接执行的纯 SQL。
- 禁止使用 PostgreSQL `DO $$ ... $$` 匿名代码块语法。

## 13. 代码可维护性
# 代码可读性强制要求
- 单个方法不超过 80 行，超过必须拆分。
- 禁止三层以上嵌套逻辑。
- 禁止魔法值，统一使用常量或枚举。
- 公共方法必须书写 Javadoc。

## 14. 通用规范
# 高频场景统一规则
- 所有列表必须分页，统一使用 `pageNum/pageSize`，空列表返回 `[]`。
- 字符串参数必须 trim，入参必须校验长度与合法性。
- 禁止 SQL 拼接，禁止使用 `.last()`/`.apply()`。
- 关键写接口必须防重复提交，不返回敏感字段。
- 库存/状态/金额等并发修改必须加锁，禁止先查后改。


## 16. 代码注释规范
所有类、接口、枚举必须添加中文注释，说明业务用途。
所有公共方法、核心业务方法必须添加中文注释，说明功能、参数含义、返回值。
核心业务逻辑、复杂判断、状态流转必须添加中文行注释说明意图。
禁止无意义注释，注释必须准确、简洁、不冗余。
英文变量 / 方法可配套中文注释，便于团队快速理解。
